Matt Pocock Skills 实战指南：让 AI 写出经得起工程审视的代码

一句话定位：Matt Pocock 的 skills 仓库是一套装进 AI 代理 .claude 目录的工程纪律——把对齐、TDD、系统调试、架构护理这些”老派软件工程”做成可重复的技能，让 Claude Code / Codex / Cursor 等代理写出经得起工程师审视的代码，而不是 vibe coding 的产物。

一、为什么这个仓库值得你花 30 分钟读完

1.1 一组震撼的数字

指标 数值 对比
GitHub Stars 166,818 比同类 SDD 框架 BMAD-METHOD（~12k）高 13 倍
Forks 14,378 —
License MIT 完全可 fork、可商用
版本 v1.1.0（2026-07-08） —
首次提交 2026-02-03 5 个月从 0 到 16.7 万 star
Newsletter ~60,000 订阅者 —

这是什么概念？这是 2026 年 AI 编程领域增速最快、最受关注的工程类开源项目。它不是一个新框架、不是新 IDE、不是新模型——它只是一些 Markdown 文件，但凝结了 Matt Pocock 几十年工程经验。

1.2 它解决了什么

Matt Pocock 在 README 里总结了 4 个 AI Coding 的核心失败模式，并对应给出解决方案：

# 失败模式 解决方案
1 Agent 没做你想要的事 /grill-me & /grill-with-docs
2 Agent 太啰嗦、术语不一致 共享语言 + CONTEXT.md
3 生成的代码不工作 /tdd + /diagnosing-bugs
4 代码库变成一团乱麻 /improve-codebase-architecture

每一项都不靠”框架接管你的开发流程”实现——Matt 明确反对 GSD、BMAD、Spec-Kit 那种”代办一切”的做法。Skills 的哲学是：

小、易适配、可组合。适用于任何模型。把控制权留给你。

二、5 分钟快速上手

2.1 安装（30 秒）

npx skills@latest add mattpocock/skills

按交互提示选择：

  1. 你要安装哪些 skills（先全选也无妨，每个都是独立模块）
  2. 装到哪个 AI 代理（Claude Code / Codex CLI / Cursor / Aider / Cline 等）
  3. 务必勾选 /setup-matt-pocock-skills——这是初始化 skill，每个仓库跑一次

2.2 初始化（每个仓库一次）

打开你的 AI 代理（以 Claude Code 为例），运行：

/setup-matt-pocock-skills

它会问你 3 个问题：

问题 默认选项 何时改
Issue tracker 用哪个？ GitHub Issues 用 Linear / Jira / 本地 Markdown 时改
Triage labels 用什么？ needs-triage / needs-info / ready-for-agent / ready-for-human / wontfix 你已有约定俗成的标签时改
Domain docs 放哪？ 单仓库布局（CONTEXT.md + docs/adr/） Monorepo 改多 context 布局

完成后会自动创建：

  • docs/agents/issue-tracker.md
  • docs/agents/domain.md
  • docs/agents/triage-labels.md（如果装了 triage skill）
  • 在 CLAUDE.md 或 AGENTS.md 加 ## Agent skills 块

2.3 第一个 Skill：和 AI 对齐需求

对齐是这套 skills 最核心、最受欢迎的入口。试试这个：

/grill-me "我想给项目加一个用户头像上传功能"

接下来 AI 会连续追问你——不接受含糊答案：

Q1: 这个功能的用户是谁？（普通用户 / 后台管理员 / 两者？）
Q2: 文件大小上限是多少？为什么是这个数字？
Q3: 支持哪些图片格式？是否需要 EXIF 处理？
Q4: 上传后的存储位置？本地？S3？CDN？
...

追问结束、AI 输出一份精炼的计划后，你的下一步才有意义。

三、核心设计哲学

3.1 仓库的二维分类

Matt Pocock 把 skills 按两个维度分类：

维度 1：调用方

类型 YAML 标记 触发方式 上下文代价
User-invoked（用户触发） disable-model-invocation: true 你输入 /skill-name 才触发 0（description 不进上下文）
Model-invoked（模型触发） 默认 AI 自动判断何时该用 占用上下文（description 一直在）

维度 2：领域

分类 数量 用途
Engineering 17 个 代码工作专用
Productivity 5 个 通用工作流（非编码）

3.2 关键约束（来自源码）

User-invoked skill 可以调用 Model-invoked skill，但不能调用另一个 User-invoked skill。这个约束保证：

  • 用户不会因级联触发失控
  • 模型每次只被一个”菜单”指挥

3.3 不接管你的流程

README 里的原话：

“Approaches like GSD, BMAD, and Spec-Kit try to help by owning the process. But while doing so, they take away your control and make bugs in the process hard to resolve.”

对比：

项目 哲学
Trellis AI 是新成员，spec 是入职手册（框架驱动）
BMAD / GSD / Spec-Kit 多角色代办一切（框架代办）
mattpocock/skills 提供工程纪律工具，你保留所有控制权（技能库）

四、4 大痛点 × 对应 Skill 详解

4.1 痛点 1：Agent 没做你想要的事 → /grill-me & /grill-with-docs

/grill-me（Productivity）

SKILL.md 全文（就是这么简洁）：

---
name: grill-me
description: A relentless interview to sharpen a plan or design.
disable-model-invocation: true
---

Run a `/grilling` session.

它本身只是个”开关”，真正工作的是底层的 /grilling 模型触发 skill——会不依不饶地追问，直到计划里每个决策分支都被覆盖。

/grill-with-docs（Engineering）

跟 /grill-me 的区别：在追问过程中顺便维护项目领域模型：

  • 精炼术语
  • 更新 CONTEXT.md（共享语言词典）
  • 把难以解释的决策记录成 ADR（Architecture Decision Records）

共享语言的力量（README 例子）

BEFORE: “There’s a problem when a lesson inside a section of a course is made ‘real’ (i.e. given a spot in the file system)”

AFTER: “There’s a problem with the materialization cascade”

22 个字 → 8 个字。每次会话、每次 prompt、每次回答都节省 token，而且让 AI 思考更准确——它不再用 20 个字描述 1 个概念。

4.2 痛点 2：Agent 太啰嗦 → CONTEXT.md + ADR

Matt 的核心洞察：冗长不是因为 AI 笨，而是因为它不知道你们项目的”行话”。

CONTEXT.md 是项目的共享词典，长这样：

# CONTEXT

## Materialization Cascade
The chain of side effects when a lesson moves from "scheduled"
to "real" (gets a slot in the file system, becomes streamable,
notifies enrolled students).

## ADR-0007: We Do Not Auto-Generate Lesson Transcripts
Decision: ...

AI 读到这个文件后，会自动用这些术语。每次都省 token，且前后一致。

4.3 痛点 3：代码不工作 → /tdd + /diagnosing-bugs

/tdd：TDD 不是仪式，是反馈循环

核心规则（来自 SKILL.md 原文）：

## Rules of the loop

- **Red before green.** Write the failing test first, then only enough code to pass it.
- **One slice at a time.** One seam, one test, one minimal implementation per cycle.
- **Refactoring is not part of the loop.** It belongs to the review stage.

## Seams — where tests go

A seam is the public boundary you test at: the interface where
you observe behavior without reaching inside.

**Test only at pre-agreed seams.** Before writing any test, write
down the seams under test and confirm them with the user.

## Anti-patterns

- Implementation-coupled — mocks internal collaborators, tests
  private methods, or verifies through a side channel
- Tautological — the assertion recomputes the expected value the
  way the code does
- Horizontal slicing — writing all tests first, then all
  implementation. Work in vertical slices instead.

关键概念”接缝（Seam）“——测试只在公共边界写，不针对内部实现。这样代码重构时测试不会大面积红。

/diagnosing-bugs：6 阶段硬核调试

Phase 1: Build a feedback loop   ← 整个 skill 的灵魂
Phase 2: Reproduce + minimise
Phase 3: Hypothesise            ← 3-5 个排序假设，不是 1 个
Phase 4: Instrument             ← 每个 probe 对应 1 个假设
Phase 5: Fix + regression test
Phase 6: Cleanup + post-mortem

Phase 1 是关键：如果有一条紧的、能复现 bug 的反馈回路，剩下都是机械工作。原文：

“Build the right feedback loop, and the bug is 90% fixed.”

它列了 10 种构造反馈回路的方法（按推荐顺序）：

  1. 失败的测试（最佳）
  2. curl / HTTP 脚本
  3. CLI + fixture
  4. Playwright / Puppeteer
  5. 重放捕获的 trace
  6. 一次性 harness
  7. Property / fuzz 循环
  8. Bisection harness（git bisect run）
  9. Differential loop（新旧版本对比）
  10. HITL bash 脚本（最后手段）

4.4 痛点 4：代码库变成一团乱麻 → /improve-codebase-architecture

这个 skill 是最有”视觉冲击力”的——它会：

  1. 派出 Explore 子代理扫描代码库
  2. 找出深度机会（deepening opportunities）——把浅模块变深的重构
  3. 生成一份自包含 HTML 报告（带 before/after 对比图）
  4. 在临时目录打开（不会污染仓库）
  5. 你选一个感兴趣的，进入 /grilling 循环深挖

报告样式（来自 SKILL.md 原文描述）：

  • Tailwind via CDN + Mermaid via CDN
  • 每个候选机会一张卡片，包含：Files / Problem / Solution / Benefits / Before-After 图 / 推荐强度（Strong / Worth exploring / Speculative）
  • 推荐强度用 badge 标识
  • 末尾有 Top recommendation

核心理念：深度模块——来自 John Ousterhout《A Philosophy of Software Design》：

“The best modules are deep. They allow a lot of functionality to be accessed through a simple interface.”

Matt 把它翻译成 AI 代理可以执行的具体词汇：模块、接口、深度、接缝、适配器、杠杆、本地性。这些词在仓库的 /codebase-design skill 中有完整定义。

6.2 Precondition：/setup-matt-pocock-skills

何时用：每个仓库第一次用 engineering skills 之前。

它做什么（来自 SKILL.md 原文）：

## Process

### 1. Explore
- git remote -v → 是不是 GitHub 仓库？
- AGENTS.md / CLAUDE.md → 有没有？
- CONTEXT.md → 有没有？
- docs/adr/ → 有没有？
- .scratch/ → 有没有？

### 2. 引导 3 个配置决策
Section A — Issue tracker  GitHub / GitLab / Local / Other
Section B — Triage labels    默认 5 个角色标签
Section C — Domain docs      单 context 或多 context（monorepo）

### 3. Confirm and edit
展示草稿，确认后再写。

### 4. Write
- 写 ## Agent skills 块到 CLAUDE.md 或 AGENTS.md
- 创建 docs/agents/issue-tracker.md
- 创建 docs/agents/domain.md
- 创建 docs/agents/triage-labels.md（如装 triage）

### 5. Done

实际产出（写完后你的仓库多出这些）：

your-repo/
├── CLAUDE.md（新增 ## Agent skills 块）
├── docs/
│   └── agents/
│       ├── issue-tracker.md    ← GitHub/GitLab/Local 行为说明
│       ├── domain.md           ← CONTEXT.md 消费者规则
│       └── triage-labels.md    ← 标签映射（如果装了 triage）

关键原则（来自 SKILL.md 原文）：

“Never create AGENTS.md when CLAUDE.md already exists (or vice versa) — always edit the one that’s already there.”

——不擅自创建文件，不破坏用户已有结构。

实际对话示例：

你：/setup-matt-pocock-skills

AI：我探测了一下这个仓库
   ✓ git remote 指向 GitHub
   ✓ 没有 CLAUDE.md / AGENTS.md
   ✗ 没有 CONTEXT.md
   ✗ 没有 docs/adr/
   ✗ 没有 .scratch/

Section A — Issue tracker
  推荐 GitHub（默认）：用 gh CLI 创建 issue
  接受吗？(yes/no)

你：yes

AI：Section B — Triage labels

/domain-modeling 的并行工作（关键细节）

/grill-with-docs 在追问时同时触发 /domain-modeling（Model-invoked），它做的事（来自 SKILL.md 原文）：

### Challenge against the glossary
当用户用词和 CONTEXT.md 冲突时立即指出：
"你的 glossary 把 cancellation 定义为 X，但你似乎是说 Y"

### Sharpen fuzzy language
当用户用模糊词时提议精确术语：
"你说 account — 是 Customer 还是 User？它们是两个不同概念。"

### Discuss concrete scenarios
对模糊的领域关系构造具体场景探测边界

### Cross-reference with code
用户说"X 工作是这样"——检查代码是否一致。
"你的代码取消整个 Order，但你刚说允许部分取消——哪个对？"

### Update CONTEXT.md inline
术语一解决立刻更新。不批量延后。

### Offer ADRs sparingly
只在三个条件都满足时提议 ADR：
1. Hard to reverse（难以撤销）
2. Surprising without context（不解释会奇怪）
3. The result of a real trade-off（真的有取舍）

ADR 的”三条件”是核心约束——不是所有决策都需要 ADR，只记录真正的、难以撤销的、有取舍的决策。这避免 CONTEXT.md 变成杂物堆。

2. Migrate the call sites over in batches (per package, per directory)

6.10 嵌入 /implement 的最后一步：/code-review

双轴审查模型（来自 SKILL.md 原文）：

Two-axis review of the diff between HEAD and a fixed point:

- Standards — does the code conform to this repo's documented
  coding standards?
- Spec — does the code faithfully implement the originating
  issue / PRD / spec?

Both axes run as parallel sub-agents so they don't pollute
each other's context, then this skill aggregates their findings.

5 步流程（精简）：

### 1. Pin the fixed point
git diff <fixed-point>...HEAD   (three-dot → merge-base)

### 2. Identify the spec source
1. Issue references in commit messages (#123, Closes #45)
2. A path the user passed
3. PRD under docs/, specs/, .scratch/

### 3. Identify the standards sources
- CODING_STANDARDS.md / CONTRIBUTING.md
- Fowler smell baseline (always applies — 12 个代码味道)

### 4. Spawn both sub-agents in parallel
- Standards sub-agent: 检查代码规范 + 12 个 Fowler smell
- Spec sub-agent: 检查是否忠实实现 spec

### 5. Aggregate
- 不要合并或 rerank findings（两个轴故意分开）
- 最后一行：每个轴的总发现数 + 最严重的问题

Fowler smell baseline（来自 SKILL.md 原文，12 个）：

Smell 简述
Mysterious Name 名字不揭示意图
Duplicated Code 同样逻辑多处出现
Feature Envy 方法更关心别人的数据
Data Clumps 几个字段总是一起出现
Primitive Obsession 用基本类型表达领域概念
Repeated Switches 同样的 switch 多处重复
Shotgun Surgery 一个改动散落在多处
Divergent Change 一个文件因多个原因被改
Speculative Generality 为不存在的需求加抽象
Message Chains a.b().c().d() 长链
Middle Man 类或函数只是 delegate
Refused Bequest 子类忽略大部分父类继承

双轴故意分开的原因（原文）：

“Reporting them separately stops one axis from masking the other.” “Code that does exactly what the issue asked but breaks the project’s conventions → Spec pass, Standards fail.”

2. Fowler "Data Clumps" →

3. Fowler "Mysterious Name" →
   src/webhooks/retry.ts:67 `dlq` should be `deadLetterWebhooks`.

   impl: src/webhooks/retry.ts:88 uses MAX_RETRIES = 3 (spec said 5).

- [ ] Red-capable — drives the actual bug code path and asserts
  the user's exact symptom
- [ ] Deterministic — same verdict every run
- [ ] Fast — seconds, not minutes
- [ ] Agent-runnable — can run unattended

- src/orders/checkout.ts

深度 vs 浅度（来自 SKILL.md 原文 ASCII 图）：

Deep module (good):
┌─────────────────────┐
│   Small Interface   │  ← Few methods, simple params
├─────────────────────┤
│                     │
│  Deep Implementation│  ← Complex logic hidden
│                     │
└─────────────────────┘

Shallow module (avoid):
┌─────────────────────────────────┐
│       Large Interface           │  ← Many methods, complex params
├─────────────────────────────────┤
│  Thin Implementation            │  ← Just passes through
└─────────────────────────────────┘

4 条原则：

1. Depth is a property of the interface, not the implementation.
   一个 deep module 内部可以有可 mock 的小零件，但它们不进 interface。

2. The deletion test.
   想象删掉这个模块——复杂度消失还是转移？
   消失 = pass-through = 删
   转移 = 还在干活 = 留

3. The interface is the test surface.
   调用者和测试穿过同一个 seam。
   想"穿过 interface 测"？那模块形状大概率错了。

4. One adapter means a hypothetical seam. Two adapters means a real one.
   不要引入只有一个 adapter 的 seam。

设计 for testability（来自 SKILL.md）：

// Testable
function processOrder(order, paymentGateway) {}

// Hard to test
function processOrder(order) {
  const gateway = new StripeGateway();
}

8.2 关键原则

原则 1：Description 是双重身份的

Model-invoked 的 description 同时承担触发条件 + 内容索引两个角色。每个字都进上下文，所以要极简：

# Good — 触发分支清晰
description: Diagnosis loop for hard bugs and performance regressions.
  Use when the user says "diagnose"/"debug this", or reports
  something broken/throwing/failing/slow.

# Bad — 重复同一个分支
description: Test-driven development. Use when the user wants to
  build features test-first, mentions test-first development,
  or wants integration tests.  # ← "test-first" 重复了

原则 2：Information Hierarchy 三级

1. In-skill step (SKILL.md 内的步骤) ← 默认
2. In-skill reference (SKILL.md 内的定义/规则)
3. External reference (用 context pointer 加载的外部文件)

push 太少 → SKILL.md 臃肿；push 太多 → agent 该读的没读到。

判断标准：branching。如果某段材料只有部分分支需要，它就该被 pointer 推出去。

原则 3：用 Leading Word 压缩

# 三处 restate
"fast, deterministic, low-overhead"  ×3 处

# 一个词替代
_tight_ loop  ← 模型已经知道"tight"意味着什么

好处：token 更少 + 模型有更准确的 hook。

原则 4：失败模式速查表

✅ 适合：

  • 你已经用 Claude Code / Codex / Cursor 等 AI 编程代理
  • 痛点是”AI 每次做出来的不对”、“代码不工作”、“项目越来越乱”
  • 想要轻量、不接管流程的方法论增强
  • 你认同”软件工程基本功 > vibe coding”

❌ 不适合：

  • 你想要全自动 AI 团队（用 BMAD / Spec-Kit）
  • 你想要强工作流约束 + 跨会话记忆（用 Trellis）
  • 你只用 IDE 自带的 AI 补全，不跟 agent 深度协作

10.2 推荐试用路径

  1. 第 1 天：装上 /grill-me 和 /tdd，在一个真实小功能上体验
  2. 第 3 天：加 /diagnosing-bugs，下次遇到诡异 bug 时用
  3. 第 1 周：跑 /improve-codebase-architecture 看 HTML 报告
  4. 第 2 周：运行 /setup-matt-pocock-skills 完成项目配置
  5. 持续：积累 CONTEXT.md，让 AI 越用越懂你的项目

10.3 不要犯的错

❌ 贪多：一次性全装 22 个 skill，但只用了 2 个——context load 是真实成本

❌ 跳过 setup：不跑 /setup-matt-pocock-skills 直接用——triage 等 skill 会找不到配置

❌ 把 skills 当框架：它们不是工作流引擎，是工具集。用哪些、什么时候用，你来定

❌ 不维护 CONTEXT.md：共享语言是这套技能的”复利”，不维护等于浪费

❌ 忽视 completion criterion：如果 skill 说”完成标准：… red-capable test exists”，不要跳过这步
