Matt Pocock Skills 完整学习路径

一句话定位：一份按 7 天 / 30 天 / 90 天节奏编排的 mattpocock/skills 学习路线图——帮你从「装上不会用」进化到「能定制、能推广给团队」，让 AI Coding 真正成为可被工程纪律约束的生产力工具。

引言：为什么是 mattpocock/skills

如果你在过去半年用过 Claude Code、Cursor、Codex 或类似的 AI Coding 工具，大概率经历过这些场景：

  • 给了 AI 一段需求，它跑得飞快，做出来的东西和你要的不一样；
  • 让 AI 修 bug，它改了五处，但只有两处真正有效，剩下三处是新引入的回归；
  • AI 帮你写完一坨代码，没有测试、没有文档、没有上下文，下次会话再问起来它什么都不记得；
  • 几个 AI sub-agent 并行跑，相互冲突、互相覆盖，最后交付的还是一个 vibe coding 的产物。

这些不是”模型不够聪明”的问题。这是流程纪律的问题。

Matt Pocock 在 2026 年初开源的 mattpocock/skills 仓库（182K stars、10.6M 安装、22 个 promoted skills），本质上是一套装进 AI 代理 .claude 目录的工程纪律。它不接管你的开发流程、不替你写代码、不承诺”全自动开发”——它只在你按下 /skill-name 时，让 AI 代理按可预测的步骤执行可预测的纪律。

仓库的核心定位在 README 里讲得很清楚：

“My agent skills that I use every day to do real engineering - not vibe coding.”

这篇文章讲”按什么顺序学、用多久、怎么验证你真的掌握了”。

我们将学习路径切成三个阶段，每个阶段都有明确的时间预算、产出标准、和升级判定条件：

阶段 时间 核心目标 产出标志
初级篇 Day 1-7 装上、能跑通第一个闭环 用 4 个 skill 完成一个真实小项目
进阶篇 Day 8-30 掌握完整工作流链 用 8 个 skill 主导一个 feature 的端到端交付
高级篇 Day 31-90+ 能定制、能推广 写出 1 个自定义 skill，在团队中标准化使用

如果你只想知道”怎么装、怎么用”，可以直接跳到 初级篇 §1.1。但如果你想真正用好这套体系，建议按本文顺序读完全文，再回到对应章节动手。

第一章：初级篇 —— 快速上手建立基础（Day 1-7）

本阶段目标：7 天内完成安装 + 初始化 + 至少 4 个核心 skill 的实战，建立”AI 是按纪律协作的伙伴”的心智模型。

1.1 30 秒安装

仓库提供两种安装路径——选择决定了你后续的定制空间：

模式 命令 文件落地 适合
skills.sh（可编辑） npx skills@latest add mattpocock/skills 文件复制到你的项目目录 想改 skill、想学习 SKILL.md 怎么写的人
Claude Code 插件（订阅式） /plugin marketplace add mattpocock/skills
/plugin install mattpocock-skills@mattpocock		 只读、自动更新 只想用、不想维护 fork 的人

初级阶段建议先用 skills.sh 模式——你能看到每个 skill 的 SKILL.md 长什么样，这是后续学习自定义 skill 的基础。

安装交互中有 3 个关键选择：

  1. 要装哪些 skills——第一次建议全选 promoted（22 个），后续按需精简；
  2. 装到哪个 AI 代理——Claude Code / Codex / Cursor / Aider 等，至少选一个你日常在用的；
  3. 是否勾选 /setup-matt-pocock-skills——必选，这是仓库级的初始化 skill。

1.2 第一次初始化：跑 /setup-matt-pocock-skills

/setup-matt-pocock-skills 是整个仓库的入口闸门——它不解决任何工程问题，它只为后续所有 engineering skills 准备好基础设施。每个新仓库只需要跑一次。

它会问你 3 个问题，并自动生成对应的项目级配置文件：

你：/setup-matt-pocock-skills

AI：探测仓库状态...
   ✓ git remote 指向 GitHub
   ✓ 没有 CLAUDE.md / AGENTS.md
   ✗ 没有 CONTEXT.md
   ✗ 没有 docs/adr/
   ✗ 没有 .scratch/

Section A — Issue tracker
  推荐 GitHub（默认）

Section B — Triage labels
  默认 5 个角色标签（needs-triage / needs-info / ready-for-agent / ready-for-human / wontfix）

Section C — Domain docs
  默认单 context：CONTEXT.md + docs/adr/ 在仓库根

3 个决策全部接受默认也能跑通整个体系。完成后你的仓库会多出这些文件：

your-repo/
├── CLAUDE.md（新增 ## Agent skills 块）
├── docs/
│   └── agents/
│       ├── issue-tracker.md    ← 告诉 AI 用哪个 tracker
│       ├── domain.md           ← 告诉 AI 怎么读写 CONTEXT.md
│       └── triage-labels.md    ← 标签映射（如装了 triage skill）
└── CONTEXT.md                  ← 项目领域术语表（初始为空）

初级阶段的核心动作：跑完这个 skill，确认生成的 3 个文件你都能看懂。如果看不懂，先读 《Matt Pocock Skills 实战》 §6.2——那里有完整的产物说明。

1.3 初级必学 4 个 skill

7 天内只需要掌握 4 个 skill 就能跑通一个有意义的闭环。少即是多——比一次性学完 22 个 skill 更重要。

Skill #1：/grill-me —— 让 AI 问你问题

触发场景：你想做一个东西，但自己还没想清楚。

典型用法：

你：/grill-me "我想给项目加一个用户头像上传功能"

AI：（连续追问 5-10 个问题）
  Q1: 这个功能的用户是谁？（普通用户 / 后台管理员 / 两者？）
  Q2: 文件大小上限是多少？为什么是这个数字？
  Q3: 支持哪些图片格式？是否需要 EXIF 处理？
  Q4: 上传后的存储位置？本地？S3？CDN？
  ...

心智模型：AI 不是你的执行者，是你的结构化思考伙伴。/grill-me 的核心是 relentlessly（不依不饶）这个 leading word——它会让 AI 拒绝接受含糊答案，直到每个决策分支都被覆盖。

Skill #2：/grill-with-docs —— 追问 + 维护领域模型

与 /grill-me 的关键区别：在追问过程中顺便维护项目领域模型——精炼术语、更新 CONTEXT.md、把难解释的决策写成 ADR（Architecture Decision Record）。

典型用法：

你：/grill-with-docs "我要做一个 webhook 重试队列，指数退避"

AI：（8-15 轮追问后）
  ## Plan: Webhook Retry Queue
    - 用 exponential backoff with full jitter
    - 最大 5 次重试
    - 失败后写入 dead_letter_webhooks 表

  ## Glossary Updates
    - 新增 "Retry Attempt" 概念
    - 新增 "Dead Letter Webhook" 概念

  ## ADR Suggestions
    - ADR-0012: Why exponential backoff with full jitter

心智模型：/grill-with-docs = /grill-me + /domain-modeling。它是主流程的入口——有 codebase 时首选它；无 codebase 时用 /grill-me。

Skill #3：/tdd —— 测试驱动开发循环

触发场景：实现功能时，AI 默认会跳到”先全部写完再说”。/tdd 强制 AI 按红绿重构循环工作。

核心规则（来自 SKILL.md 原文）：

## Rules of the loop
- Red before green.   写失败的测试，然后只写够让它通过的代码
- One slice at a time.   一个 seam、一个测试、一个最小实现
- Refactoring is not part of the loop.   它属于 review 阶段

## Seams — where tests go
- Test only at pre-agreed seams.   测试只在预先商定的 seam
- Before writing any test, write down the seams under test and confirm

心智模型：seam 是你和 AI 事先约定的测试边界——而不是”AI 自己觉得应该测哪里”。这就是为什么 /to-spec 必须先和用户确认 seams。

Skill #4：/diagnosing-bugs —— 6 阶段调试纪律

触发场景：bug 抵抗第一眼判断（间歇性、性能回退、依赖多）。

6 阶段流程：

Phase 1: Build a feedback loop   ← 整个 skill 的灵魂
Phase 2: Reproduce + minimise
Phase 3: Hypothesise            ← 3-5 个排序假设，不是 1 个
Phase 4: Instrument             ← 每个 probe 对应 1 个假设
Phase 5: Fix + regression test
Phase 6: Cleanup + post-mortem

铁律：“No red-capable command, no Phase 2.”——没有可复现的命令，不进入假设阶段。这条规则防止最常见的调试失败模式：直接跳到假设、修改代码、祈祷它工作。

心智模型：bug 难修不是因为”找不到原因”，而是因为没有可重复的反馈回路。Phase 1 一旦建成，剩下都是机械工作。

1.4 第一个完整闭环：3 个 skill 串起来用

光知道 4 个 skill 是不够的——你必须把它们串起来跑一遍才能形成肌肉记忆。下面是一个 30 分钟的练习项目：

目标

用 mattpocock/skills 做一个命令行 todo app（10-15 个文件规模的小项目）。

完整流程

Step 1：/grill-me "我想做一个命令行 todo app"
   → 让 AI 问清楚需求（存储位置、命令格式、并发模型）

Step 2：/grill-with-docs "把刚才的计划 + 加上领域模型"
   → 产出 ADR + 更新 CONTEXT.md

Step 3：/implement（AI 会自动调用 /tdd）
   → AI 按红绿重构循环实现

Step 4：（完成后）/code-review
   → 双轴审查（Standards + Spec）

期望时间：30-60 分钟（含 AI 跑测试时间）。

验证标志：

4 个常见踩坑

踩坑 1：在 Step 1 中，AI 给的计划你已经满意，但还是接受了。

修正：/grill-me 的目的是让你想清楚，而不是让 AI 给答案。如果你对答案满意，说明问得不够深——回到 Step 1 继续追问。

踩坑 2：Step 2 直接复用 Step 1 的对话，跳过了 grill-with-docs。

修正：/grill-with-docs 会同时跑 /grilling 和 /domain-modeling——这两件事必须在一个会话里同时发生，否则术语和决策会脱节。

踩坑 3：Step 3 中，AI 没有按红绿循环工作，而是先写了一堆代码再补测试。

修正：明确告诉 AI “按 /tdd 的规则，每个 seam 一个测试一个最小实现”。如果 AI 还是跳步骤，重新发起 /implement 并加约束。

踩坑 4：跳过 Step 4 的 code-review。

修正：code-review 是 /implement 的收尾必选项——SKILL.md 里明确写了 “Once done, use /code-review to review the work”。不要相信”我自己看一眼就行”。

1.5 初级阶段的产出标准

完成 Day 7，你应该能：

能力 验证方式
在 30 秒内独立完成安装 关掉所有笔记，凭记忆操作
解释 3 个核心配置文件的作用 issue-tracker.md / domain.md / triage-labels.md 各自管什么
独立选择 4 个 skill 的触发场景 给出 4 个真实场景，正确匹配到 /grill-me / /grill-with-docs / /tdd / /diagnosing-bugs
完成至少 1 个完整闭环 用 3 个 skill 串起来做一个 todo app / calculator / JSON parser 等

如果有一项不达标——重复对应章节的练习，直到能凭记忆复述核心步骤。

第二章：进阶篇 —— 效率提升，工作流优化（Day 8-30）

本阶段目标：从”会用 4 个 skill”升级到”掌握完整工作流链 + 协作技能”，能主导一个中型 feature 的端到端交付。

2.1 进阶篇的整体地图

初级篇学的 4 个 skill 都是单点能力。进阶篇的核心是把它们串成有向图——/ask-matt skill 里的完整路由地图是这样的：

   ▼                     ▼                     ▼
/triage              /diagnosing-bugs       /wayfinder
(堆积的 issue)         (修 bug)              (超大任务)
   │                     │                     │
   └─────────────────────┼─────────────────────┘
                         │

            │  主流程：idea → ship     │
            │                          │
            │  /grill-with-docs  ★入口 │
            │     ↓                   │
            │  [可选]/prototype 绕行   │
            │     ↓                   │
            │  /to-spec               │
            │     ↓                   │

健康通道（独立运行）/improve-codebase-architecture（每周）
词汇层（被上层调用）/domain-modeling /codebase-design
跨会话桥接 /handoff
Standalone /grill-me /prototype /research /teach

进阶篇的核心任务：把这条主流程跑通 3-5 次，直到每个 skill 在何时该用、何时不该用形成直觉。

2.2 主流程第一步：/to-spec —— 把对话沉淀为规格

初级阶段你学会了 /grill-with-docs 输出计划。但计划是对话产物，不持久、不可被其他人/agent 引用。/to-spec 把对话转化为结构化 spec，发到 issue tracker。

关键能力：选择测试 seam。

/to-spec 的核心动作之一是 “sketch out the seams at which you’re going to test the feature”。原则：

优先级 规则
1 优先用现有 seam
2 最高层 seam 优先
3 seam 越少越好（理想是 1 个）
4 必须和用户确认

seam 一旦在 spec 阶段确定，/tdd 和 /code-review 都围绕这些 seam 工作——这是为什么 spec 阶段必须严肃对待 seam 选择。

Spec 模板（来自 SKILL.md 原文）：

## Problem Statement
The problem that the user is facing, from the user's perspective.

## Solution
The solution to the problem, from the user's perspective.

## User Stories
A LONG, numbered list of user stories:
1. As a <actor>, I want a <feature>, so that <benefit>

## Implementation Decisions
- The modules that will be built/modified
- The interfaces of those modules that will be modified
- Technical clarifications from the developer
- Architectural decisions
- Schema changes
- API contracts

## Testing Decisions
- What makes a good test (only test external behavior)
- Which modules will be tested
- Prior art for similar tests

## Out of Scope
A description of the things that are of scope.

## Further Notes

关键约束（来自 SKILL.md 原文）：

“Do NOT include specific file paths or code snippets. They may end up being outdated very quickly.”

但有例外——如果 prototype 产出了能比文字更精确表达决策的代码片段（state machine、reducer、schema、type shape），可以内联到相关 decision 并注明来源。

2.3 主流程第二步：/to-tickets —— spec 拆为可执行切片

spec 是文档，没法直接执行。/to-tickets 把 spec 拆为垂直切片（vertical slices）——每个切片都端到端可演示、可验证。

核心概念：tracer bullet（示踪子弹）——先打穿一条窄路径，每个切片都穿过所有层（schema、API、UI、tests）。

## 3. Draft vertical slices

Break the work into tracer bullet tickets.

- Each slice cuts a narrow but COMPLETE path through every layer
  (schema, API, UI, tests) — vertical, NOT a horizontal slice
  of one layer
- A completed slice is demoable or verifiable on its own
- Each slice is sized to fit in a single fresh context window
- Any prefactoring should be done first

给每个 ticket 标注 blocking edges（这是与单纯 todo 的本质区别）：

# 03 — Add exponential backoff calculation

**What to build:** End-to-end working exponential backoff with
full jitter, verifiable via CLI test.

**Blocked by:** None — can start immediately.

**Status:** ready-for-agent

- [ ] Backoff calculator unit tests pass
- [ ] CLI demo shows 100 random backoff values within expected bounds
- [ ] README updated with formula

关键词：work the frontier——任何”blockers 都完成”的 ticket 都可立即开始。

Wide refactor 的例外

/to-tickets 默认强制垂直切片，但有一种例外：

“A wide refactor is one mechanical change — rename a column, retype a shared symbol — whose blast radius fans across the whole codebase. Don’t force it into a tracer bullet; sequence it as expand-contract: Expand → Migrate → Contract.”

工程现实里，重命名、类型同步这种跨切片工作不能套 tracer bullet——这时用 expand-contract 模式：先加新形式、再迁移调用点、最后删旧形式。

2.4 主流程第三步：/implement —— 驱动 /tdd + /code-review + commit

/implement 是主流程的执行器。它的 SKILL.md 全文就这么简洁：

---
name: implement
description: "Implement a piece of work based on a spec or set of tickets."
disable-model-invocation: true
---

Implement the work described by the user in the spec or tickets.

Use /tdd where possible, at pre-agreed seams.

Run typechecking regularly, single test files regularly, and the
full test suite once at the end.

Once done, use /code-review to review the work.

Commit your work to the current branch.

关键约束：

  • “Use /tdd where possible, at pre-agreed seams”——不是”全部用 TDD”，而是”在 spec 里约定的 seam 上用 TDD”
  • “Run full test suite ONCE AT THE END”——不要每写一个文件都跑全套
  • “Once done, use /code-review”——code-review 是收尾必选项
  • “Commit your work to the current branch”——默认 commit 到当前分支，不开新分支

关键经验

/ask-matt 原文：

“Keep steps 1–3 (grilling, spec, tickets) in one unbroken context window — don’t compact or clear until after /to-tickets — so the grilling, spec, and tickets all build on the same thinking.”

但每个 /implement 都开新会话——避免上下文污染。

2.5 进阶阶段必学的 4 个新 skill

主流程上你已经会用 /grill-with-docs（初级篇）+ /to-spec + /to-tickets + /implement（进阶篇）。下面是主流程之外的 4 个关键 skill：

Skill #5：/code-review —— 双轴审查

/code-review 不是普通的代码审查——它分两个独立轴，并行运行、分开报告：

         ┌─────────────────┐
         │   code-review   │
         └─────────────────┘
              │
       ┌──────┴──────┐
       ▼             ▼
   Standards        Spec
   (规范+smell)    (需求忠实度)
轴 检查内容
Standards 编码规范、Fowler smells 基线、命名约定、风格一致性
Spec 是否忠实实现原始 issue/PRD、范围是否蔓延、需求完整性

关键设计：两个 sub-agent 并行审查，不让一个轴掩盖另一个轴的问题。原文：

“Reporting them separately stops one axis from masking the other.”

Fowler smell baseline（12 个代码味道）：Mysterious Name、Duplicated Code、Feature Envy、Data Clumps、Primitive Obsession、Repeated Switches、Shotgun Surgery、Divergent Change、Speculative Generality、Message Chains、Middle Man、Refused Bequest。

Skill #6：/diagnosing-bugs 深度应用

初级篇你已经知道 6 阶段流程。进阶篇的核心是真正用 Phase 1 走出死循环。

Phase 1 的 4 个 checkbox：

- [ ] Red-capable — drives the actual bug code path and asserts
  the user's exact symptom
- [ ] Deterministic — same verdict every run
- [ ] Fast — seconds, not minutes
- [ ] Agent-runnable — can run unattended

10 种构造反馈回路的方法（按推荐顺序）：

  1. 失败的测试（最佳）
  2. curl / HTTP 脚本
  3. CLI + fixture
  4. Playwright / Puppeteer
  5. 重放捕获的 trace
  6. 一次性 harness
  7. Property / fuzz 循环
  8. Bisection harness（git bisect run）
  9. Differential loop（新旧版本对比）
  10. HITL bash 脚本（最后手段）

进阶心智：如果 Phase 1 找不到合适的回路，这本身就是一个发现——SKILL.md 原文：

“If no correct seam exists, that itself is the finding. The codebase architecture is preventing the bug from being locked down. Flag this for the next phase.”

找不到测试 seam = 代码架构有问题，应转给 /improve-codebase-architecture。

Skill #7：/triage —— issue 状态机

作为维护者，面对一批还没处理的 GitHub Issues / GitLab Issues 时用。

两类角色 + 五种状态：

## Roles
Two category roles:
- bug — something is broken
- enhancement — new feature or improvement

Five state roles:
- needs-triage       — maintainer needs to evaluate
- needs-info         — waiting on reporter for more information
- ready-for-agent    — fully specified, ready for an AFK agent
- ready-for-human    — needs human implementation
- wontfix            — will not be actioned

每个被 triage 的 issue 应该有恰好一个 category role + 一个 state role。

透明性纪律：每个 AI 在 triage 期间发的评论必须以 *This was generated by AI during triage.* 开头——让 reporter 知道这是 AI 生成的评论。

Skill #8：/handoff —— 跨会话上下文交接

何时用：

  • 当前会话上下文快满了
  • 需要开个新会话但保留上下文
  • prototype 绕行后需要回到主流程

关键约束（来自 SKILL.md 原文）：

  • 写到 OS 临时目录（macOS $TMPDIR，Linux /tmp，Windows %TEMP%）—— 不污染工作区
  • 不重复已存在 artifact（spec/plan/ADR/issue/commit/diff）—— 用路径引用
  • 自动脱敏（API key、密码、PII）
  • 自动生成 “suggested skills” 段，提示下个 agent 该用哪些 skill

典型产物：

# Handoff — Webhook Retry Queue Implementation

## TL;DR
Design phase complete. PRD + 2 ADRs + 1 prototype variant validated.
Next session should start with /to-spec on the PRD, then /to-tickets.

## What's Done
- /grill-with-docs: 8 questions answered, plan finalized
- /prototype: state machine validated for 5 cases (PROTO-001)
- CONTEXT.md updated with Retry Attempt + Dead Letter Webhook
- ADR-0011: Exponential backoff with full jitter

## What's Next
1. /to-spec (formalize the plan into a structured spec)
2. /to-tickets (break into tracer-bullet tickets)
3. /implement per ticket

## Suggested Skills for Next Session
- /to-spec → /to-tickets → /implement (each ticket clears context)
- /tdd during /implement (at the pre-agreed seams)
- /code-review at the end

## Redacted
- Customer API keys (3 occurrences)
- Internal service URLs (replaced with placeholder.com)

2.6 CONTEXT.md 与 ADR 的工程纪律

CONTEXT.md 的本质：项目共享词典——不是 spec，不是 implementation log，不是 todo list。

# CONTEXT

## Materialization Cascade
The chain of side effects when a lesson moves from "scheduled"
to "real" (gets a slot in the file system, becomes streamable,
notifies enrolled students).

## ADR-0007: We Do Not Auto-Generate Lesson Transcripts
Decision: ...

AI 读到这个文件后会自动用这些术语——每次都省 token，且前后一致。

ADR 的三条件（核心约束）：

Offer ADR 仅在三条件都满足时提议 ADR： 1. Hard to reverse（难以撤销） 2. Surprising without context（不解释会奇怪） 3. Real trade-off（真的有取舍）

反模式：把所有决策都写成 ADR。这会让 ADR 变成杂物堆，掩盖真正重要的决策。

2.7 进阶阶段的常见瓶颈

瓶颈 1：AI 不按 /tdd 循环工作

症状：AI 先写了一堆代码再补测试，或者一次写多个测试再写多个实现。

诊断：

  • 你是否在 /to-spec 阶段明确指定了 seam？
  • 你是否在 /implement 调用时显式说”按 /tdd 的规则工作”？

修正：

/implement .scratch/webhook-retry/issues/01-create-retry-attempt.md
  --constraint "按 /tdd 的红绿重构循环，一个 seam 一个测试"
  --constraint "refactoring 不属于循环"

瓶颈 2：/code-review 报告混淆两个轴

症状：AI 把规范问题和需求偏差混在一起报告，标准问题”喧宾夺主”。

诊断：你可能在调用时没说”分两轴独立报告”。

修正：

/code-review
  --constraint "Standards 轴和 Spec 轴分开报告，不要合并或 rerank"
  --constraint "每个轴独立打分"

瓶颈 3：CONTEXT.md 越写越乱

症状：CONTEXT.md 里有实现细节、临时笔记、过期 ADR，最后变成杂物堆。

诊断：你可能在 /grill-with-docs 时没有及时清理 CONTEXT.md——很多术语一次性写入，没有删除/合并。

修正：跑一次 /domain-modeling（model-invoked），让它自己主动挑战和清理过时的术语。原文：

“Challenge against the glossary”——“你的 glossary 把 cancellation 定义为 X，但你似乎是说 Y”

瓶颈 4：/triage 把所有 issue 都标记为 ready-for-agent

症状：很多 issue 实际上需要人类判断（产品决策、跨团队协调），但 AI 全部打 ready-for-agent。

诊断：你没有给 AI 清晰的边界。

修正：在 setup 阶段就预先定义每个状态对应的处理规则——例如 “ready-for-agent 必须有完整 spec，没有 spec 必须转 needs-info”。

2.8 进阶阶段的产出标准

完成 Day 30，你应该能：

能力 验证方式
跑通完整主流程 /grill-with-docs → /to-spec → /to-tickets → /implement → /code-review
选择合适的 seam 给定一个 feature，能列出 3 个候选 seam 并解释取舍
双轴审查的配置 强制 Standards 和 Spec 分开报告，能识别常见的合并错误

处理 issue 队列 用 /triage 处理 10 个 issue，准确率 > 80%

3.2 编写自定义 SKILL.md

SKILL.md 的目录结构

your-skill-name/
├── SKILL.md              # 必需：定义文件
├── agents/
│   └── openai.yaml       # 可选：Codex UI 元数据
└── *.md                  # 可选：参考文档

YAML Frontmatter 规范

---
name: skill-name                    # 必需：kebab-case slug
description: 一句话触发描述         # 必需：说明何时使用
disable-model-invocation: true      # 可选：true=仅用户触发
argument-hint: "提示文字"           # 可选：参数占位提示
---

Description 的编写要点

好的 description 应包含：

  1. 触发场景：什么情况下使用
  2. 核心机制：做什么（动词）
  3. 关键约束：避免误用

反例：

description: Improve code quality.

问题：太抽象，AI 无法判断何时触发。

正例：

description: Test-driven development. Use when the user wants to build
  features or fix bugs test-first, mentions "red-green-refactor", or wants
  integration tests.

包含：触发场景（用户想用 TDD）、核心机制（红绿重构）、关键约束（只在预先商定的 seam）。

双轴分类：何时设 disable-model-invocation

场景 设置 理由
编排型 skill（如 /implement、/grill-me） true 仅用户主动触发，避免 AI 误触发
可复用纪律（如 /tdd、/diagnosing-bugs） 不设置 用户和模型都可触发，让 AI 在合适场景自动选用

Leading Words 设计

每个 skill 必须有一个核心词汇作为模型思考的锚点：

Skill Leading Word
tdd red-green-refactor
diagnosing-bugs tight feedback loop
codebase-design deep module
grilling relentless interview

用法：在 SKILL.md 的开头和每个章节反复出现这个词汇，让模型围绕它思考。

渐进式披露原则

信息组织应该分层：

层级 用途 位置
In-skill steps 主流程的有序步骤 SKILL.md 正文
In-skill reference 需时查阅的定义 SKILL.md 末尾或附录
External reference 独立深度内容 单独的 .md 文件

反模式：把所有内容塞进一个长文档，导致 context window 浪费。

3.3 编写自定义 Skill 的实战示例

下面是一个真实场景——你团队有一个强制 PR 必须跑过所有测试的纪律，可以写一个 /check-tests-passed skill：

Step 1：识别触发场景

  • 用户提交 PR 后希望自动验证测试是否全部通过
  • 用户说”跑测试”、“CI 通过”、“所有测试通过”

Step 2：设计 SKILL.md

---
name: check-tests-passed
description: Verify all tests pass before merging a PR. Use when the
  user says "run tests", "verify CI", "check tests passed", or wants to
  confirm the PR is mergeable.
disable-model-invocation: true
argument-hint: "[PR number or branch]"
---

# Check Tests Passed

## When to use
- User wants to verify a PR is mergeable
- User asks for CI status

## Process

### 1. Identify the PR
- If PR number provided: `gh pr view <number>`
- If branch provided: `gh pr list --head <branch>`
- Otherwise: check current branch

### 2. Check CI status
- `gh pr checks <number>` — list all checks
- For each check: get state (success / failure / pending)

### 3. Run local test suite (if no CI configured)
- `pnpm test` (or appropriate command from package.json)
- Report pass/fail count

### 4. Aggregate result
- All green → "✅ All tests passed. PR is mergeable."
- Any red → "❌ <N> checks failed: <list>"

## Anti-patterns
- Don't run tests if user hasn't approved the cost (large test suite)
- Don't modify code — only verify

Step 3：测试与迭代

测试你的自定义 skill：

  1. 创建一个 PR（不通过测试）
  2. 运行 /check-tests-passed
  3. 验证 AI 正确报告失败
  4. 修复测试，重新运行
  5. 验证 AI 报告成功

迭代要点：

  • AI 经常”自由发挥”——明确边界（don’t modify code）
  • AI 经常忽略细节——明确格式（for each check, get state）
  • AI 经常超时——明确步骤（if CI configured, just check）

3.4 团队级部署：Plugin vs Skills.sh

把 mattpocock/skills 推广给团队时，需要决定走哪条路径：

维度 skills.sh Claude Code Plugin
文件落地 复制到每个仓库 全局只读、自动更新
定制空间 可 fork、可修改 不可修改
更新策略 手动 rebase 自动
适用场景 团队希望 fork + 改 skill 团队只想用、不想维护 fork
协作成本 高（每个仓库要更新） 低（更新一次全员受益）

推荐策略

Hybrid 模式：

  1. 核心 skills 用 Plugin 模式——所有团队成员共享，自动更新；
  2. 团队特定的 skills 用 skills.sh 模式——复制到团队的 .claude/skills/ 目录，可定制；
  3. 关键文档（CONTEXT.md 模板、ADR 模板）放在仓库内——每个项目独立维护。

这种模式兼顾标准化和灵活性。

推广节奏

第 1 周：选 1 个试点项目，完整跑通主流程
第 2 周：收集痛点，编写团队特定的 skill（如 check-tests-passed）
第 3 周：在 3-5 个项目推广，记录效果
第 4 周：开内部培训，把核心 4-6 个 skill 教给所有工程师

3.5 双轴 code-review 的深度配置

进阶篇你已经会用 /code-review。高级篇的核心是把它工程化——配置成 PR 流程的一部分。

配置方案

# .claude/skills/code-review/SKILL.md（自定义扩展）

## Repo-specific Standards
- TS strict mode: 必须
- Test coverage: >= 80% for new code
- PR description must reference issue number
- Breaking changes must update ADR

## Repo-specific Spec sources
- PRD: docs/prd/<feature>.md
- Issue tracker: GitHub Issues
- Linked issues from commit message

把 repo-specific 的规范写进 SKILL.md——AI 每次审查时都会先读这个文件。

自动化触发

在 CI/CD 中加入 code-review 触发：

# .github/workflows/ai-review.yml
name: AI Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run /code-review
        run: |
          claude-code /code-review \
            --pr-number ${{ github.event.number }} \
            --output review-${{ github.event.number }}.md
      - name: Post comment
        uses: actions/github-script@v6
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync(`review-${{ github.event.number }}.md`, 'utf8');
            github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: review
            });

双轴审查的常见问题与修正

问题 1：Spec 轴报告”diff 没实现 issue 提到的 X 功能”，但实际 diff 里有 X

通常是 AI 没找到 issue 引用。修正：在 commit message 里强制要求 Refs #<number>。

问题 2：Standards 轴报告”命名不规范”，但项目没文档化命名规范

修正：先把命名规范写进 CONTRIBUTING.md，让 AI 有标准可循。

问题 3：两个轴都通过了，但代码就是有 bug

这是双轴审查的已知边界——它检查的是规范符合性和需求忠实度，不检查逻辑正确性。逻辑正确性需要 /tdd + 人类审查覆盖。

3.6 Wayfinder：大规划能力

/wayfinder 处理任务规模超过一个 agent 会话能装下的场景。它把任务规划为决策地图，每张票问一个决策。

关键概念

概念 定义
Map 一个 wayfinder:map 标签的 issue，是 index，不是 store
Tickets map 的子 issue，每张票问一个决策
Destination 这张地图最终要到达什么
Fog of war 你知道有但还没法精确描述的决策
Frontier 已开、未阻塞、未被认领的票——地图的”活边缘”

Ticket 类型

Type 谁驱动 用例
Research (AFK) Agent 后台读 调研 API、读文档
Prototype (HITL) 人类 验证”应该长什么样”
Grilling (HITL) 人类对话 决策类问题
Task (HITL or AFK) 任方 必须做某事才能继续决策

关键纪律

“Never resolve more than one ticket per session.”——一个会话一张票，保持推理质量。

“Wayfinder is planning by default: each ticket resolves a decision, and the map is done when the way is clear — nothing left to decide before someone goes and does the thing.”

核心区别：Wayfinder 的产物是决策，不是代码。它处理的是”还没想清楚要做什么”，而不是”想清楚了怎么实现”。

3.8 高级阶段的常见瓶颈

瓶颈 1：自定义 skill 写得过于复杂

症状：SKILL.md 超过 200 行，包含大量分支条件。

诊断：你可能在试图让一个 skill 做太多事。

修正：拆分。每个 skill 单一职责。如果一个 skill 同时处理”测试 + 部署 + 监控”，拆成 3 个 skill。

瓶颈 2：团队成员不遵守 skill 调用纪律

症状：工程师绕过 /tdd 直接写代码，或者跳过 /code-review。

诊断：纪律问题，不是工具问题。

修正：

  • 在 PR 模板里强制要求引用用过的 skill（如 “This PR was developed with /grill-with-docs + /implement”)；
  • 在 CI 中强制检查 PR 是否经过 /code-review；
  • 在 code review 中强制检查 测试覆盖 seam 是否与 spec 一致。

瓶颈 3：CONTEXT.md 与 ADR 增长失控

症状：一年后 CONTEXT.md 有 500 行，ADR 有 50 个，新成员 onboarding 痛苦。

诊断：缺乏定期清理机制。

修正：

  • 每季度一次 CONTEXT.md 审计——删除过时术语、合并重复概念；
  • 每年一次 ADR 归档——把”已稳定 1 年以上”的 ADR 移到 docs/adr-archive/，不删除（保留历史可追溯）；
  • 新成员 onboarding 时强制读 CONTEXT.md——把”必须先读 CONTEXT.md 才能改这个项目”写入 CONTRIBUTING.md。

3.9 高级阶段的产出标准

完成 Day 90+，你应该能：

能力 验证方式
编写自定义 skill 写一个 50-100 行的 SKILL.md，能通过双轴审查
设计团队级部署方案 给出混合模式（Plugin + skills.sh）的具体配置
双轴 code-review 工程化 在 CI/CD 中集成，能产出可重复的报告
Wayfinder 决策地图 为一个中型 feature 创建 map + 10+ tickets
企业流程集成 issue tracker + CI/CD + 知识管理都有对应方案
团队推广能力 能向新成员培训核心 4-6 个 skill，并解释设计哲学

持续的工程纪律

仓库的核心哲学：

“A skill exists to wrangle determinism out of a stochastic system.”

Skill 的价值不在于产生相同输出，而在于可预测的流程。模型是随机的，skill 是把随机性驯化成确定性的工具。

最重要的不是学多少 skill，而是养成”按纪律工作”的习惯。即使没有 AI agent，下面这些原则仍然适用：

  • 测试只在预先商定的 seam
  • 调试前先建立反馈回路
  • 代码审查分规范和需求两个轴
  • 重大决策写 ADR，但限制数量
  • 项目有自己的领域词典，持续维护

AI 只是放大器——它放大你的纪律，也放大你的混乱。
